# rig — deployment walkthrough & session context

> Paste this to initialize a new session. It briefs you on `rig` so you can help debug a from-scratch
> deployment to an NVIDIA Jetson Orin (images pulled from a local registry on the dev box). The user drives
> the steps and will call on you only if they hit an issue.

## What rig is

`rig` is a vehicle/machine-level sensor-stack orchestrator: **"a loop + a manifest"** that *delegates*
bring-up/teardown of each stack to that service's own per-sensor launcher (`<service>-up`). It owns only the
cross-cutting concerns (instance-name uniqueness, fleet ROS env, ordering, status, lifecycle, deployment
artifacts) and never reimplements per-stack logic. Dependency is one-way: rig depends on the service repos
via their `rigging.yaml` descriptor + launcher CLI; a service never knows about rig. The rig tool lives at
`/Users/ckt/ws/bringup` (git, **v0.1.8**, Python 3 + PyYAML). It's run-from-source: `./rig <verb>`.

Each **service repo** carries its rig integration: a root `rigging.yaml` descriptor + a `<service>-up`
launcher + `docker/compose.deploy.yaml` + a render helper (`tools/render_params.py`, or the camera
service's `tools/sensor_env.py`). The thin drivers are generated by a shared Copier template
(`/Users/ckt/ws/boilerplate`); the camera repo (`camera-service`) is bespoke.

## Mental model — the files in a deployment

- **`vehicle.yaml`** (the manifest): `vehicle` name, `vehicle_id` (decides the ROS domain + exported as
  `VEHICLE_ID`; explicit `ros.domain_id` overrides), `ros: {rmw, distro}`, `images: {registry}`,
  **`infra:`** (shared services brought up FIRST / torn down last, e.g. a zenoh router), **`sensors:`**.
- **`services.yaml`** (the catalog): `service -> { path: <repo> }` (relative to the deployment dir).
- **`config/sensors/<name>.yaml`** + **`config/infra/<name>.yaml`**: one config per instance. May be a full
  named config OR a **nameless profile** reused across instances via a per-entry `overrides:` patch.
- rig reads only `service` + `name` from a config; everything else is **opaque** (the launcher reads it).
- **Fleet env** rig exports to every launcher: `ROS_DOMAIN_ID`, `RMW_IMPLEMENTATION`, `VEHICLE_ID`,
  `RIG_IMAGE_REGISTRY`. Each compose opts in (e.g. `image: ${NOVATEL_IMAGE:-${RIG_IMAGE_REGISTRY:+...}/novatel_driver:latest}`).
- The **launcher contract**: `<service>-up <config> [up [-d]|down|status|logs|config]`, accepts a config at
  any host path, honors fleet ROS env, machine output on **stdout** / human lines on **stderr**.

## Commands

`rig init <dir>` · `doctor` · `up [names] [--dry-run]` · `down [names] [--purge]` · `status [-v]` ·
`logs <name> [-f]` · `config [names]` · `vendor <svc> --from <repo>` ·
`bake --tag <t> [--registry <host>]` · `unbake <artifact> [--into <dir>]`. Run from the deployment dir
(rig auto-detects it by `vehicle.yaml` in the cwd) or pass `--root <dir>`.

## The walkthrough (user drives)

> The concrete, copy-pasteable runbook for the 2-camera (USB + RTSP) + dashboard deployment — including the
> exact configs, `images.tag` (JetPack), and the rig-vs-compose-only note — is **`docs/RUNBOOK.md`**. The
> generic shape is below.

```bash
mkdir ~/rig-walkthrough && cd ~/rig-walkthrough
git clone <novatel> novatel ; git clone <sbg> sbg_driver ; git clone <camera-service-github-url> camera-service
git clone /Users/ckt/ws/bringup rig
alias rig="$HOME/rig-walkthrough/rig/rig"

rig init my-vehicle && cd my-vehicle
# author services.yaml (paths -> ../<repo>, + zenoh-router -> ../rig/templates/zenoh-router),
#        vehicle.yaml (vehicle_id, rmw: rmw_zenoh_cpp, images.registry, infra: [zenoh-router], sensors:),
#        config/sensors/*.yaml  and  config/infra/zenoh-router.yaml
rig doctor                      # validate (unique names, one distro, launchers present, zenoh guardrail)
rig up --dry-run                # see exact launcher invocations + fleet env (infra first)
rig vendor camera-service --from ../camera-service   # + novatel, sbg, zenoh-router
# --- deploy ---
docker push devbox:5000/<repo>:<tag>          # push your arm64 images to the local registry (all of them)
rig bake --registry devbox:5000 --tag test1   # -> var/artifacts/test1.tar.gz (digest-pinned)
scp var/artifacts/test1.tar.gz orin:/tmp/
# on the Orin: rig unbake /tmp/test1.tar.gz --into /opt/rig && cd /opt/rig/test1 && ./run.sh up
```

## Gotchas / current state

- **Multiple instances of one service** (e.g. two cameras): each manifest entry needs a **unique `name`** —
  rig keys the compose project (`<service>_<name>`), ROS namespace (`/<name>`), and shm volume off it, so
  instances don't collide. Because everything is **host networking**, set any human/consumer-facing port
  (WebRTC signalling, preview) **distinct per instance**; `rig doctor` flags a clash if the service declares
  `host_ports`. Author them as two separate configs, or a nameless profile + per-instance `overrides:`. The
  launcher reads the source/connection per config — rig is source-agnostic (passes the config through).
- **PyYAML is installed globally on this Mac** (`/opt/homebrew/bin/python3`), so `./rig` and the launchers
  work directly — **no venv, no `*_PYTHON` exports**. On the Orin: `sudo apt install python3-yaml`.
- **camera service**: repo `camera-service` (cloned from GitHub), service `camera-service`, launcher
  `cam-up` (all renamed from the old gige-vision-service/gige-up). Descriptor file is `rigging.yaml`
  (legacy `deploy.yaml` still accepted as a fallback).
- **rmw_zenoh needs a zenoh router** (one per vehicle) in `infra:`. A ready template ships at
  `rig/templates/zenoh-router/` — point `services.yaml` at it + add an `infra:` entry. **Tune its image/
  command** for the real rmw_zenoh router (default `eclipse/zenoh`; `ros2 run rmw_zenoh_cpp rmw_zenohd` on a
  ROS image is the recommended alternative). `rig doctor` warns if rmw is zenoh but no router is declared.
- **Local registry / offline:** `rig bake --registry <host>` digest-pins (via `docker buildx imagetools`).
  Images must be **arm64** and pushed to the registry. The Orin must **trust** the registry —
  `insecure-registries: ["devbox:5000"]` in `/etc/docker/daemon.json` + restart docker (or TLS). After the
  first pull images cache locally → the vehicle runs offline.
- **The Mac CAN run containers** for functional/integration tests (arm64-native, host networking works,
  real images present), but NOT production fidelity — PTP hardware timestamps, `--device` passthrough,
  NVENC, real load. Those need the Orin. The camera's *production* path needs the NVIDIA runtime (Jetson
  has it; the Mac doesn't — rig doesn't pass cam-up's `--dev` flag yet).
- The artifact's **compose-only** form runs with just Docker (`./run.sh` uses rig if Python+PyYAML present,
  else the static `up.sh`/`down.sh`/`status.sh`).

## What's built vs not

**Built:** lifecycle (up/down/status/logs/config/doctor) · overrides + nameless profiles · `infra:` tier ·
`vehicle_id`→domain + `VEHICLE_ID` · fleet image registry (`RIG_IMAGE_REGISTRY`) · `vendor` (files + dirs) ·
`rig build` (per-service build + mirror) · `bake`/`unbake` + `--registry` digest pinning + compose-only form ·
`init` + cwd detection · `rigging.yaml` descriptor (deploy.yaml fallback). 24 tests (`tests/`, run with `python3 tests/test_*.py`).

**Not yet:** `bake --bundle-images` (full air-gap, no registry) · OCI artifact format · ROS `/diagnostics`
as a second status layer · a boot-time systemd unit · full host-facing-port-clash extraction ·
`rig adopt`/`verify` for non-template repos · the zenoh-router image/command is a template to finalize.
